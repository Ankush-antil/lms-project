Complete Technical Guide

Web-Based Audio Calling System

Hinglish Documentation (Developer Friendly)

1. Introduction

Yeh document humare LMS-Project ke Web-Based Audio Calling Feature ka complete structural and architectural explanation hai. Is feature ki help se Students (including Guest Students) aur Teachers online real-time voice call par connect ho sakte hain. Saath hi agar koi user offline hai, toh call automatically database me store ho jaati hai aur user ke wapas online aane par use Missed Call Notification Dashboard me dikhti hai.

2. Key Technologies (Kis Kis Ka Use Kiya Hai)

Is project me humne pure audio and signaling architecture ko handle karne ke liye majorly niche di gayi technologies ka use kiya hai:

1. Socket.IO (Signaling Server)

Real-time communication, events transmission (call offers, answers, ICE candidates) aur live user status tracking (Online/Offline status notifications) ke liye use kiya hai.

2. WebRTC (P2P Audio Call)

Browser ki native RTCPeerConnection API ka use kiya hai. Isse audio data bina kisi central media server ke, direct peer-to-peer (P2P) sound card se travel karta hai, jisse zero latency multi-channel connection banta hai.

3. Web Audio API (Tone Synthesizer)

Bina kisi .mp3 ya external audio assets file ke, direct Javascript code ki help se browser sound output chip par sound waves generate karke cheerfully dual-tone ringtones banata hai.

4. React Context API (Global State)

Poore client app ko single SocketProvider context me wrap kiya hai, taki page dynamic transitions ya redirect hone par bhi call hold rahe aur call screen modal overlay har page par visible ho ske.

5. Node.JS & Express API

Socket instance aur Express server ko merge karne ke liye http.createServer ka utilize kiya hai, saath me missed calls list aur call read/delete APIs ko secure and public controllers me serve kiya hai.

6. MongoDB & Mongoose

Hanging or offline calls details ko save karne ke liye CallLog schema ka build kiya hai, jisme caller reference, receiver reference aur dynamic guest attributes save hote hain.

3. WebRTC & Socket.IO Signaling (Kasa Kasa Kam Karta Hai)

WebRTC direct point-to-point audio send karne me tab tak capability nahi rakhta jab tak dono sides (peers) ko ek dusre ke Network Network Details (IP, Ports) aur Audio Formats ka pta na ho. In details ko exchange karne ke process ko Signaling kehte hain, jise humne Socket.IO se handle kiya hai:

A. Dialing & Registration Phase

1
User Registration
Jaise hi koi User login karta hai ya Guest student mail enter karke test screen me enter karta hai, client socket connect hota hai aur server par register event send karta hai. Server socket.id ko database userId se map kar leta hai.
2
Calling Request
Student call button par click karta hai. Client mic permission access karta hai aur internal SDP (Session Description Protocol) Offer create karta hai. Socket connection ke zariye server ko call-user event send hota hai.
3
Online / Offline Route Checks
Server check karta hai ki kya target receiver online hai. Agar offline hai, toh database me status missed save ho jata hai aur student ko user-offline alert milta hai. Agar online hai, toh database me call entry initiated banti hai aur target socket ko incoming-call signal forward ho jata hai.

B. Call Connection Phase

4
Offer & Answer Exchange
Call accept hone par, receiving browser apna audio set karke SDP Answer generate karta hai aur server ke through caller browser ko call-accepted signal bhejta hai. Dono side SDP receive hone par peer description confirm hoti hai.
5
ICE Candidate Exchange
RTCPeerConnection direct connect karne ke liye paths (ICE Candidates) fetch karta hai. Har path socket ke through automatically transparently pass hota hai. Dono browsers me se jo sabse fastest path hota hai use connect kar diya jata hai.
6
P2P Audio Connection
Dono end par stream dynamically attach ho jati hai, timers chalne lagte hain aur call voice state me transparently live run hone lagti hai (Vite-Client built index code par).

4. Web Audio API Synthesizer (Ringtone Engine)

Bina kisi local file (mp3) load kiye hum sound waves generate kar rahe hain. Iska syntax low-level design par audio chips se frequencies control karta hai:

// Dual-tone generator for calling UI const playTone = () => { const audioCtx = new (window.AudioContext || window.webkitAudioContext)(); const osc1 = audioCtx.createOscillator(); const osc2 = audioCtx.createOscillator(); const gainNode = audioCtx.createGain(); osc1.type = 'sine'; osc2.type = 'sine'; if (isRingback) { // Standard US Dial Tone (440Hz + 480Hz) osc1.frequency.value = 440; osc2.frequency.value = 480; gainNode.gain.setValueAtTime(0, audioCtx.currentTime); gainNode.gain.linearRampToValueAtTime(0.15, audioCtx.currentTime + 0.1); } else { // Cheerful Incoming Ring (660Hz + 880Hz) osc1.frequency.value = 660; osc2.frequency.value = 880; gainNode.gain.setValueAtTime(0, audioCtx.currentTime); gainNode.gain.linearRampToValueAtTime(0.25, audioCtx.currentTime + 0.15); } osc1.connect(gainNode); osc2.connect(gainNode); gainNode.connect(audioCtx.destination); osc1.start(); osc2.start(); };

5. Missed Calls & Guest Identification Logic

Humare is calling system me Guest Students (jinke paas backend validation JWT token nahi hai) unke status validation aur data mismatch ko prevent karne ke liye niche likhe checks perform hote hain:

  • Guest Identity Format: Guest user ki email space replace karke dynamic ID generate hoti hai (e.g. guest_ankush_gmail_com).
  • Non-Blocking Database Schema: Database schema `CallLog.js` me caller field optional hai aur additional guestName aur guestEmail attributes hain.
  • Server Hook Checks on Disconnect: Server side par socket disconnection event par async query chalti hai:
    const activeCalls = await CallLog.find({ $or: [ { caller: mongoose.Types.ObjectId.isValid(userId) ? userId : null }, { receiver: mongoose.Types.ObjectId.isValid(userId) ? userId : null }, { guestEmail: userId.toString().startsWith('guest_') ? userId.toString().replace('guest_', '') : null } ], status: { $in: ['initiated', 'connected'] } });
    Is query se, active caller agar tab click / window exit karta hai toh status database me end ho jata hai aur system block nahi hota.

6. Call Status States

Humne application me multi-state indicators design kiye hain:

  • idle - Koi call inactive hai. Overlay display hide rehta hai.
  • dialing - Caller ringback sound sunta hai, signal path build ho rha hai.
  • incoming - Receiver screen popup screen par cheerful ringtone bajti hai.
  • connected - Call connect ho chuki hai. Mute aur Live visualizer soundwave indicators active rehte hain.
  • offline / declined / ended - Call end hone ke baad state clean updates ke liye clear animation display hoti hai (2.5 seconds).
